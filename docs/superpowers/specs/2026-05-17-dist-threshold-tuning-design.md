# DIST_THRESHOLD の値設定 設計（C 後半）

- 日付: 2026-05-17
- 対象: 乗務地図関係 / トラッカーのマッチ許容半径 `DIST_THRESHOLD` を実測ジッターに基づき適正化する
- 前提 spec: `2026-05-16-tracker-jitter-instrumentation-design.md`（G-4＝C 前半）、`2026-05-16-tracker-stall-roi-restriction-design.md`（G-2）

## 背景

G-2 の根本原因調査で、`DIST_THRESHOLD=0.06`（フレーム間マッチの許容半径、正規化座標）が駐車車の間隔（約 0.035 正規化）より大きく、ある車の検出が一時的に欠けたとき隣の車に誤マッチしうることが判明した。C は「ジッター実測 → 適正値設定」の2段で、G-4（C 前半）で `update_tracks` のマッチ距離を `matched_dists` として `vehicle-track-history.jsonl` に記録し始めた。本タスク（C 後半）はその実測データから `DIST_THRESHOLD` の値を決める。

## 実測データ分析

G-4 デプロイ（2026-05-16 15:49）以降、約16時間ぶん・`matched_dists` 3154 サンプル（914 行、real01_line + real02）。

| 区間下端 | 件数 | 累積% |
|---|---|---|
| 0.000 | 2627 | 83.3 |
| 0.005 | 185 | 89.2 |
| 0.010 | 69 | 91.3 |
| 0.015 | 56 | 93.1 |
| 0.020 | 36 | 94.3 |
| 0.025 | 35 | 95.4 |
| 0.030 | 18 | 95.9 |
| 0.035 | 33 | 97.0 |
| 0.040〜0.055 | 95 | 100.0 |

0〜0.005 に 83%（静止した駐車車）。累積 91% が ≤0.015、94% が ≤0.025、96% が ≤0.035。0.035〜0.06 に約4%のテール。

### カメラ別

| カメラ | n | p90 | p95 | p99 |
|---|---|---|---|---|
| real02（stall4） | 2351 | 0.0049 | 0.010 | 0.050 |
| real01_line（stall1+2+3） | 803 | 0.0333 | 0.044 | 0.056 |

- real02 はジッター極小（p90 < 0.005）。駐車車がほぼ完全静止。
- real01_line はマッチ距離が大きく広がる（p90 0.033、駐車間隔 0.035 の直前）。

## 解釈

- 両カメラとも**支配的モード（main cluster）は near-zero**。real02 はほぼ全体、real01_line も約64%（≈511/803）が ≤0.005。spec（G-4 設計）§④の escalation 条件「main cluster が 0.035 を超える」には該当しない → 単一閾値で対応可能。
- real01_line の 0.02〜0.06 の広がりは、(a) 現 `DIST_THRESHOLD=0.06` が駐車間隔 0.035 より広いため貪欲マッチが隣の車の検出に誤マッチした距離が記録された、(b) stall1+2+3 の3 stall ぶんで車が多く動きも多い、の混合。
- 0.035 を超えるマッチ距離は駐車間隔を超えており、静止駐車車のジッターではない（隣車への誤マッチ、または出庫中の移動車）。閾値で切り捨てるのが正しい。
- `matched_dists` は現 `DIST_THRESHOLD=0.06` で左側打ち切り（0.06 超は未マッチで記録されない）。検討対象の閾値は ≤0.035 で 0.06 を十分下回るため、打ち切りは分析に影響しない。

## 決定: `DIST_THRESHOLD` を 0.06 → 0.025

G-4 設計 §④の判定基準「main cluster の上端の直上 ∧ 駐車間隔 0.035 未満」に従う:

- プール累積 94% が ≤0.025 — main cluster の裾を覆う。
- 駐車間隔 0.035 に対し 0.01 のマージン下 — 隣の車への誤マッチ（≈0.035 付近）を排除する。
- real02（クリーンなカメラ）には p95 0.010 に対し 2.5 倍の余裕。
- 0.06 からの中庸な引き下げ。`MAX_MISSED=2` が一時的な取りこぼし（あるカメラの car が単発で 0.025 を超えるジッターを起こした場合）を吸収する。

0.02〜0.03 の範囲はいずれも 0.06 からの厳密な改善だが、0.025 を選ぶ（main cluster の裾を覆い、かつ駐車間隔に 0.01 のマージンを残す中庸点）。

## 実装

`scripts/track_vehicles.py` の定数:

```python
DIST_THRESHOLD = 0.06
```

を:

```python
DIST_THRESHOLD = 0.025
```

に変更する（コメントがあれば実測ジッター由来である旨に更新）。

- `update_tracks` は純関数で、マッチ許容半径 `dist_threshold` を引数で受ける。ロジックは不変。
- 定数 `DIST_THRESHOLD` は `main()` が `update_tracks` を呼ぶときに渡される値。
- `tests/test_track_vehicles.py` の `update_tracks` テストは閾値を明示的な引数（`0.06`）で渡しており、定数 `DIST_THRESHOLD` に依存しない。よってテストは不変（変更しない。テストはマッチアルゴリズムを閾値非依存に検証するもの）。

## スコープ外（後続・要再測定）

- **デプロイ後の再測定**: 0.025 適用後、`matched_dists` を再度蓄積して real01_line のマッチ距離分布を確認する。分布が締まれば（0.02〜0.06 の広がりは 0.06 閾値由来の誤マッチだった）C 完了。締まらず 0.03 付近に厚いままなら、real01_line は真にジッター/動きが大きい → 貪欲＋距離マッチ方式を IoU ベース等に変える別タスクへエスカレーション（G-4 設計 §④のエスカレーション節）。
- **per-camera 閾値**: real02 と real01_line で特性が異なるが、単一 `DIST_THRESHOLD=0.025` で両対応可（real02 は余裕大、real01_line は妥当）。per-camera 別閾値化は本タスクでは行わない（YAGNI、必要なら将来）。
- `update_tracks` のマッチアルゴリズム（貪欲＋距離）の変更。

## 後続検証（C 再測定）— 2026-05-17 完了・クリーン確定

`DIST_THRESHOLD=0.025` デプロイ（`de11256`、2026-05-17 07:36 JST 稼働）後の `vehicle-track-history.jsonl`（schema 3）から、デプロイ時刻以降の `matched_dists` を集計（166 tick）:

| カメラ | n | p50 | p90 | max | ≤0.025 |
|---|---|---|---|---|---|
| real01_line | 464 | 0.0008 | 0.0038 | 0.0235 | 100% |
| real02 | 941 | 0.0007 | 0.0029 | 0.0248 | 100% |

**判定: 分布は締まった。C 完了。**

- G-7 前の real01_line は p90≈0.033（0.06 閾値由来の誤マッチで 0.02〜0.06 に裾が広がっていた）。0.025 適用後は p90=0.0038 と約10倍締まり、両カメラとも全サンプルが 0.025 以内。
- スコープ外節の「締まらず 0.03 付近に厚いままなら IoU マッチへエスカレーション」の条件には**該当しない**。`update_tracks` の貪欲＋距離マッチ方式は real01_line/real02 双方で妥当に機能しており、**IoU ベースへの変更タスクは不要**。
- 集計窓は約3時間（166 tick / 1405 サンプル）と短いが、分布は境界事例ではなく明確（100% が閾値の 1/6 以下）。数日窓での再確認も結論を覆さない見込み。
