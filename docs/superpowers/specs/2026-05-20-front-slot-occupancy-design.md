# 乗り場 先頭スロット占有方式 設計書

> 作成: 2026-05-20

## 目的

乗り場別の出庫計測を、YOLO トラッカー方式から「**各乗り場の先頭領域のスロット占有を
画像解析で数え、その時系列の減少を出庫として積算する**」方式へ置き換える。

## 背景・問題

旧方式（YOLO トラッカー）は、密集した黒いタクシーを YOLO が取りこぼし、さらに ROI が
斜めの待機レーンに合わず、乗り場別の出庫が不正確だった（乗り場3≈0・乗り場1過剰）。
ROI 校正を繰り返したが、検出と1台ずつの追跡そのものが密集環境で不安定。

## 羽田空港 第1待機所の運用ルール（計測の前提）

東京タクシーセンターの第1待機所（国内線プール）。ユーザー（現役乗務員）の説明と
公式レーン図に基づく:

- 各乗り場、**先頭の横1列**が並ぶ。画像で**奥から順に出庫**していく。
- 先頭列が抜けると、**後続列が前へ詰める＝列移動**（プールのタクシーも前へ詰める）。
- プールが混雑（タクシーの増加 > 利用者）した時は、あふれ防止で**余計に一列詰める**
  動きもある。
- **容量（WEB調査）**: 乗り場1＝8台、乗り場2＝7台、乗り場3＝8台。乗り場4は**不均一**で
  先頭4台・後続は8台や7台が並ぶ列もある。
- **入庫はプール後方**で起きる。したがって**先頭領域の台数は「出庫（減る）」と
  「列移動の補充（増える）」でしか変化しない** — 入庫の影響を受けない。

## 採用アプローチ

各乗り場の「**先頭領域**」をスロット格子で定義する。各スロットの在/不在を**画像解析**
（エッジ密度・黒率）で判定し、乗り場ごとに先頭領域の**在台数**を毎 tick 数える。
在台数の時系列で「**減少**」を出庫として積算する（小さな1台ずつの減少＝出庫。大きな
増加＝列移動の補充なので出庫に数えない）。

なぜ先頭領域だけか: 入庫はプール後方で起きるため、先頭領域に絞れば台数変化が
出庫と列移動の2要因に限定され、「入庫と出庫が相殺して取りこぼす」問題が起きない。

不採用:
- YOLO トラッカー（密集黒車の検出漏れ・1台追跡の不安定・ROIずれ）。
- プール全体の占有差分（入庫と出庫が相殺し出庫を取りこぼす）。

## 設計

### 1. スロット定義（新スキーマ `stall-slots.json`）

各乗り場の先頭領域を、車1台ぶんの**スロット**の格子で定義する。

```json
{
  "_meta": { "image_size": [800, 600], "note": "スロット中心は 0-1 正規化座標" },
  "schema_version": 1,
  "stalls": {
    "stall1": { "source": "real01_line", "label": "第1乗り場 ...",
                "slots": [ {"id": "1-1", "cx": .., "cy": .., "r": ..}, ... ] },
    "stall2": { "...": "..." },
    "stall3": { "...": "..." },
    "stall4": { "...": "..." }
  }
}
```

- `slots[*]`: スロット1個＝先頭領域の駐車1マス。`cx,cy`＝中心（正規化）、`r`＝判定円の
  半径（正規化、または小矩形の半幅）。
- 容量: 乗り場1/2/3 は先頭1〜2列ぶん（8/7/8 幅）、乗り場4は不均一（先頭4＋後続8/7）。
  実際のスロット数・位置は校正で確定する。
- 各乗り場の slots は画像で**奥→手前の順**に並べる（出庫順＝奥からに対応）。

### 2. 在/不在判定（画像解析）

- 各スロットの判定領域（小円/小矩形）の画素から、エッジ密度・黒率・輝度を算出する
  （`scripts/lib/image-pool-analyzer.mjs` の `analyzeROI` 相当）。
- しきい値で在（車あり）/不在（空きアスファルト）を判定する純関数
  `slot_occupied(features, threshold)`。
- 空きアスファルトは低エッジ・均一、車は高エッジ。先頭列は車が分離して写るため
  密集プールより判定しやすい。
- しきい値は時間帯（昼/夕/夜）で見え方が変わるため、**時間帯適応または校正で調整**する。
  初期は固定しきい値＋実画像チューニング、必要なら輝度連動。

### 3. 出庫カウント（在台数の差分）

- 各 tick（速め＝60秒間隔）、乗り場ごとに先頭領域の在台数 `occ` を数える
  （= 占有スロット数）。
- 在台数の時系列で**減少**を出庫として積算する純関数:
  - `occ` が前 tick より減った → 減少分を出庫に加算。
  - `occ` が増えた → 列移動の補充。出庫に数えない。
  - 検出フリッカ対策のデバウンス（1 tick だけの揺れを無視 or N tick 確定）。
- 乗り場別の出庫数を 15分スロット等に集計する。

### 4. 出力

- 乗り場別の出庫数を、既存の到着便ページ「実績／予測」が読む形式で出力する
  （`departedByStall` 相当のスキーマを流用）。下流の予測パイプライン・日報アプリは
  原則不変。
- track-history（YOLO 由来）は廃止または併存（実装計画で決定）。

### 5. 校正

- 校正支援ツールで、各乗り場の先頭領域のスロット格子を定義する（カメラ画像にスロットを
  重ねて確認・調整）。
- 在/不在しきい値を実画像（昼・夜）で確認・調整する。

## データフロー

```
カメラ画像 → 各スロット領域を画像解析 → スロット在/不在 → 乗り場別の在台数 occ
  → occ の時系列差分（減少＝出庫・増加＝列移動）→ 乗り場別出庫数 → 実績/予測
```

## テスト方針（TDD）

- `slot_occupied(features, threshold)`: 既知のエッジ/黒率で在/不在を返す純関数。
- 在台数差分→出庫積算の純関数: 減少（出庫）・増加（列移動）・フリッカ（デバウンス）の
  各ケース。
- スロット定義の読み込み（`stall-slots.json` パース）。
- しきい値校正は手動・実画像で担保。

## スコープ外

- プール全体の待機台数計測（本件は先頭領域＝出庫計測が対象）。
- 旧 YOLO トラッカー資産の撤去手順の詳細（実装計画で扱う）。
- カメラ画角変更時の再校正自動化。

## 成功基準

- 乗り場別の出庫が、先頭領域スロット占有の時系列差分で計測される。
- `slot_occupied`・出庫積算ロジックの単体テストがパス。
- 実データで乗り場別出庫が観測実態（奥から順・列移動）と整合する。
- 到着便ページの実績／予測が乗り場別に出る。
