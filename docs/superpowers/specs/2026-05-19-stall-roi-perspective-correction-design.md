# 乗り場ROIの台形補正（ホモグラフィ）設計書

> 作成: 2026-05-19

## 目的

乗り場別出庫計測のROI（判定領域）を、カメラの斜め撮影による透視ゆがみに対応させる。
ホモグラフィ（台形補正）でカメラ画像の地面を真上視点へ変換し、その座標系で
乗り場ROIを定義することで、出庫の乗り場別計測を正確にする。

## 背景・問題

`stall-rois.json` の乗り場ROIは、画像右端の真っ直ぐな縦帯（x600〜800）に固定された
軸並行矩形で、2026-05-12に手描きされたまま「実装時に微調整」が一度もされていない。

実カメラ画像で検証した結果（`/tmp/stall-roi-check.png`）:
- 駐車場の車は画像の左〜中央に密集し、待機列はカメラの斜め撮影で**斜めに伸びている**。
- 乗り場2・乗り場3のROIは車のいない空きアスファルトの上にある → 計測ほぼ0。
- 乗り場1のROIだけ列の上端をかすめて拾う → 過剰計測。
- 実測3フレームで「ROI内に入った車は乗り場1の2台だけ、乗り場2・3はゼロ」。

配信データでも乗り場1≫乗り場3（約14:1、乗り場3はほぼ0）の偏りが確認された。
軸並行矩形では斜めの待機列を表現できないことが根本原因。

## 採用アプローチ

**点変換方式のホモグラフィ。** カメラ画像の地面平面を真上視点（鳥瞰）へ写すホモグラフィ
行列をカメラごとに求め、検出した車の点をその行列で真上座標へ変換してから乗り場ROI
（真上座標系の長方形）で判定する。

- **画像は変形しない。** YOLO検出は元画像のまま実行する。画像自体を真上視点に歪めると
  遠方の車が引き伸ばされYOLOの検出精度が落ちるため。変換するのは「点」だけ。
- カメラは固定なので、ホモグラフィの校正は一度行えば恒久的に使える。

不採用:
- **軸並行矩形ROIの引き直し** — 斜めの待機列を矩形で近似できず、同じ問題を繰り返す。
- **画像変形方式（真上視点画像でYOLO検出）** — 検出精度劣化。YOLOは通常視点で学習。
- **元画像上の多角形ROI** — 真上座標系で長方形を引くより、歪んだ画像上で四隅を
  目分量で置くほうが間違えやすい（ROIがズレ続けてきた根本原因がこれ）。

## 設計

### 1. カメラ校正モデル（`stall-rois.json` 新スキーマ v2）

カメラごとにホモグラフィの基準点と、乗り場ROIを真上視点座標で保持する。
旧スキーマ（歪んだ縦帯の軸並行矩形）は廃止。

```json
{
  "_meta": { "...": "..." },
  "schema_version": 2,
  "cameras": {
    "real01_line": {
      "image_size": [800, 600],
      "reference_points": [[x1,y1],[x2,y2],[x3,y3],[x4,y4]],
      "rectified_size": [RW, RH],
      "rectified_corners": [[0,0],[RW,0],[RW,RH],[0,RH]]
    },
    "real02": { "...": "同様" }
  },
  "stalls": {
    "stall1": { "source": "real01_line", "capacity": 8, "label": "...",
                "rect": { "x": ..., "y": ..., "width": ..., "height": ... } },
    "stall2": { "source": "real01_line", "...": "..." },
    "stall3": { "source": "real01_line", "...": "..." },
    "stall4": { "source": "real02",      "...": "..." }
  }
}
```

- `reference_points`: 元画像（ピクセル座標）で、現実では長方形になっている地面上の
  4点。順序は `rectified_corners` と対応（左上→右上→右下→左下）。
- `rectified_corners`: その4点が真上視点キャンバスで対応する4隅。`rectified_size` の
  矩形の四隅。
- ホモグラフィ `H` = `reference_points`（正規化: ÷image_size）→ `rectified_corners`
  の透視変換。`cv2.getPerspectiveTransform` で構築する。
- `stalls[*].rect`: 真上視点座標系での乗り場領域（軸並行長方形）。真上視点では
  駐車区画が格子状になるため長方形で十分。

### 2. ホモグラフィ・点変換（`track_vehicles.py`）

- カメラごとに、起動時に `reference_points`（正規化）→ `rectified_corners` から
  ホモグラフィ行列 `H` を構築する。
- 純関数 `to_rectified(x, y, H)`: 正規化済みの元画像座標の点を `H` で真上視点座標へ変換。
  `cv2.perspectiveTransform` 相当（3×3行列と斉次座標の積）。
- ホモグラフィの入力は **正規化された元画像座標**（0〜1）。`track_vehicles.py` の
  既存処理が正規化座標で動くため整合させる。`H` 構築時に `reference_points` を
  `image_size` で割って正規化しておく。

### 3. 変換に使う点＝バウンディングボックス下端中央

ホモグラフィは**地面平面上の点**に対して正確。車体中心は地面より上に浮くため、
車のタイヤ接地点に近い **bounding box の下端中央**（`x = (x_min+x_max)/2`,
`y = y_max`）を、検出点・追跡点・乗り場判定点として一貫して使う。

- `detect_vehicles` は bounding box を持つ。`track_vehicles.py` 側で各検出を
  下端中央の点に変換してから追跡に渡す。
- 追跡（`update_tracks`）は従来どおり**正規化された元画像座標**で行う（下端中央の
  点を使う点だけが変更）。距離しきい値 `DIST_THRESHOLD` のスケールは元画像正規化座標
  のままなので不変。ホモグラフィ変換は ROI 判定時にのみ適用する。

### 4. ROI判定（`track_vehicles.py`）

- `filter_to_rois(detections, ...)`: 各検出点（下端中央）を `to_rectified` で真上座標へ
  変換し、そのカメラの全乗り場ROI（長方形）の和集合内にあれば残す。**戻り値は元の
  （正規化元画像座標の）検出点**＝追跡は従来座標系で行う。
- `stall_of_point(x, y, ...)`: トラックの最後位置（正規化元画像座標）を `to_rectified`
  で変換し、真上座標でどの乗り場ROI長方形に入るかを返す。どこにも入らなければ None。
- 出力 `cameras[*].departedByStall`（track-history schema v4）は不変。下流（throughput
  校正・実績 `computeTrackActuals`・予測・日報アプリ）は一切変更不要。

### 5. 校正プロセスと校正スクリプト

カメラ固定のため校正は一度きり。新規 Python スクリプト
（`scripts/calibrate-perspective.py`、OpenCV 使用）で支援する。既存の
`calibrate-stall-rois.mjs`（旧グリッド表示・旧スキーマ前提）は本作業で役割を終える。

校正の流れ（カメラごと: real01_line, real02）:

1. **基準4点の決定** — 現在フレームを取得し、駐車場の白線で現実では長方形になる
   地面上の4隅を選ぶ。スクリプトは候補4点をオーバーレイした注釈画像を出力し、
   ユーザーが「その4点でよい／ここを直す」と確認する。
2. **真上視点画像の生成** — 確定した4点からホモグラフィを構築し、元画像を真上視点へ
   ワープした確認用画像（グリッド付き）を出力する。歪みが取れて駐車区画が格子状に
   見えることを確認する。
3. **乗り場ROIの指定** — 真上視点画像（グリッド付き）を見て、ユーザーが乗り場1〜4の
   領域を指定する。スクリプトは指定領域をオーバーレイした注釈画像を出力し、合うまで
   調整する。
4. **書き出し** — 確定した `reference_points`・`rectified_size`・各乗り場 `rect` を
   `stall-rois.json`（新スキーマ）に書き出す。

校正スクリプトは「現在フレーム取得」「基準点から真上視点画像を生成」「ROIを
オーバーレイした注釈画像を生成」「新スキーマJSONを書き出す」を担う。基準点・ROIの
座標は引数または対話で受け取る。座標値そのものはユーザーと注釈画像を見ながら詰める。

### 6. データフロー

```
元画像 → YOLO検出（bbox）→ 下端中央の点（正規化元画像座標）
  → filter_to_rois（H変換して真上座標でROI和集合内か判定・元座標で残す）
  → update_tracks（正規化元画像座標で追跡）
  → トラック消失時 stall_of_point（H変換して真上座標でどの乗り場か）
  → departedByStall（schema v4・不変）
```

### 7. テスト方針（TDD）

- `to_rectified`: 既知のホモグラフィ（既知の基準4点）で、基準点が
  `rectified_corners` に写ること、内分点が期待座標に写ることを検証。
- `stall_of_point`: 既知のホモグラフィ＋乗り場長方形で、真上座標で各ROI内に写る点が
  正しい乗り場名を返し、ROI外が None を返すことを検証。
- `filter_to_rois`: 真上座標でROI和集合内の検出を残し、外を除外。戻り値が元の
  正規化元画像座標であることを検証。
- bbox→下端中央の変換関数の単体テスト。
- 校正スクリプトは手動・一度きりのため自動テスト対象外。`node --check` 相当の
  構文確認と実フレームでの目視確認で担保。
- Python 全テスト回帰（`tests/test_track_vehicles.py` ほか）。

## スコープ外

- **YOLOの検出漏れ** — 密集した黒いタクシーを取りこぼす（実測で50台中11〜14台のみ
  検出）。ホモグラフィでは直らない別問題。本設計でROIを正しくした後、実測で深刻さを
  評価し、必要なら別途対応する。
- カメラのレンズ歪み（樽型歪み）の補正。ホモグラフィは平面射影のみ扱う。実用上、
  乗り場4領域の分類には平面射影で十分とみなす。
- 乗り場をまたぐ車両の高度な追跡。タクシー乗り場では発生しない。
- `arrived`（入庫）の扱い。本件は出庫計測のROIが対象。

## 波及・確認事項

- `stall-rois.json` のスキーマが v1→v2 に変わる。`stall_rois_for_camera` を読む箇所は
  `track_vehicles.py` のみ（確認済み）。校正スクリプト旧版は新版へ置き換え。
- track-history schema v4（`departedByStall`）は不変。下流リポジトリ・日報アプリは
  無変更。
- 校正の基準点・ROI座標はユーザーと注釈画像を見ながら確定する。設計段階では
  「校正の仕組み」を作り、実際の座標値は校正実施時に決める。

## 成功基準

- `stall-rois.json` が v2（カメラごとのホモグラフィ基準点＋真上視点ROI）になる。
- `track_vehicles.py` が点変換ホモグラフィで乗り場判定する。
- 既知ホモグラフィでの `to_rectified`・`stall_of_point`・`filter_to_rois` の単体
  テストがパス。Python 全テスト回帰なし。
- 校正後の実データで、乗り場別出庫が観測実態と整合する（乗り場3がほぼ0でなくなる、
  乗り場1の過剰計上が解消する）。
