# 羽田空港 到着便ビューワー — 設計ドキュメント

- **作成日**: 2026-04-25
- **ステータス**: 設計承認待ち
- **対象ユーザー**: 1人（作成者自身、タクシー乗務員）
- **公開方式**: Public GitHub Pages（既存タクシー乗務アプリの第3タブとして追加）
- **関連既存設計**: `2026-04-24-taxi-ic-helper-design.md`（タイマー + IC判定）

---

## 1. 目的

羽田空港の **到着便スケジュール + 推定降客数** を時間帯別に可視化し、タクシー乗務中に「いま羽田に向かうべきか／どのターミナルが混むか」を判断するための情報源とする。

### ユースケース（優先順）

1. **A. リアルタイム判断**（主用途）
   - 直近〜数時間先の到着便を確認し、空港待機 / 出庫タイミングを決める
2. **B. パターン把握**（副次）
   - 時間帯別ピークを目視し、シフト計画の参考にする

### スコープ

- **対象空港**: 羽田空港（HND）の **到着便のみ**（出発便は対象外）
- **対象ターミナル**: T1（JAL系国内線）／T2（ANA系国内線）／T3（国際線）
- **時間ウィンドウ**: 過去30分 〜 未来3時間（5分ごと更新）
- **1人用アプリ**: 認証なし、自分専用

### 非スコープ

- 出発便、第3ターミナル以外の国際線拠点
- ユーザー登録、通知機能、お気に入り便
- 到着遅延予測（ODPTのestimatedTimeをそのまま表示）

## 2. データソース（確定）

### 2.1 ODPT（公共交通オープンデータセンター）

**用途**: 到着便スケジュール + リアルタイム到着情報

| 項目 | 内容 |
|---|---|
| エンドポイント | `https://api.odpt.org/api/v4/odpt:FlightInformationArrival` |
| 認証 | 開発者登録（無料）→ アクセストークン |
| データ形式 | JSON-LD |
| カバー | JAL / ANA + 一部LCC（5社目安） |
| 更新頻度 | リアルタイム（数分ごと） |
| ライセンス | 公共交通オープンデータ基本ライセンス（出典明記必要） |

#### 取得フィールド

| フィールド | 用途 |
|---|---|
| `odpt:flightNumber` | 便名（"JL123" など） |
| `odpt:airline` / `odpt:operator` | 航空会社 |
| `odpt:departureAirport` | 出発空港（IATAコード） |
| `odpt:terminal` | T1/T2/T3 |
| `odpt:scheduledTime` | 定刻 |
| `odpt:estimatedTime` | 予定到着時刻（遅延反映） |
| `odpt:actualTime` | 実到着時刻 |
| `odpt:flightStatus` | 定刻/遅延/到着など |
| `odpt:aircraftModel` | 機材コード（**null混在あり**） |

### 2.2 国土交通省 航空輸送統計

**用途**: 路線別平均搭乗率（搭乗者数推定の根拠）

| 項目 | 内容 |
|---|---|
| 出典 | 国土交通省「航空輸送統計年報」「航空輸送サービスに係る情報公開」 |
| データ形式 | 公表PDF / 月次データ |
| 取得方法 | 手動で年1回程度マスタ更新 |
| 内容 | 路線×月別の平均搭乗率（%） |

#### 取得粒度（運用上の妥協）

- 月次更新ではなく**年次平均**でマスタ化（路線別 1値）
- 主要路線を網羅: 羽田-伊丹 / 羽田-千歳 / 羽田-福岡 / 羽田-那覇 / 羽田-旭川 など20-30路線
- 小規模路線は **70%（国内線平均）** をフォールバック値として使用

### 2.3 機材→座席数マスタ

**用途**: 機材コードから座席数を引く

| 項目 | 内容 |
|---|---|
| 出典 | JAL/ANA公式サイト（機材紹介・座席表ページ） |
| 形式 | `data/aircraft-seats.json`（手動編集） |
| 内容 | 機材コード → 座席数（国内線仕様優先）|

#### 主要機材一覧（実装時に確定）

| コード | 機材 | 座席数（国内線仕様） |
|---|---|---|
| B789 | Boeing 787-9 | 246 |
| B788 | Boeing 787-8 | 200 |
| B772 | Boeing 777-200 | 405 |
| B773 | Boeing 777-300 | 525 |
| B738 | Boeing 737-800 | 166 |
| A320 | Airbus A320 | 146 |
| A321 | Airbus A321neo | 194 |
| A359 | Airbus A350-900 | 369 |
| DH8D | Bombardier Q400 | 74 |
| ... | （計20-30機材） | ... |

### 2.4 採用しなかった案（記録）

| 案 | 採用しなかった理由 |
|---|---|
| JAL/ANA予約状況スクレイピング | **両社規約で明示的に禁止**。ANA「ロボット・スパイダー等で監視またはコピー禁止」。JAL「アグリゲーション技術での転載禁止」。民事・刑事リスクあり |
| FlightAware / FlightRadar24 API | 有料、無料枠ではタクシー用途に不十分 |
| 出発空港HPのスクレイピング | 各空港が**そもそも搭乗者数を保有していない** |
| GDS（Amadeus等）API | 法人契約のみ。個人開発不可 |

## 3. アーキテクチャ

### 3.1 全体構成

```
┌──────────────────────────────────────────────┐
│  GitHub Actions (cron: */5 * * * *)         │
│  - ODPT API → 到着便取得                     │
│  - aircraft-seats.json + load-factors.json  │
│    で搭乗者数推定                             │
│  - data/arrivals.json を生成・commit         │
└────────────────┬─────────────────────────────┘
                 │ git push
                 ▼
┌──────────────────────────────────────────────┐
│  GitHub Pages (static)                       │
│  - index.html (タイマー)                     │
│  - ic.html (IC判定)                          │
│  - arrivals.html (新規)                      │
│  - data/arrivals.json (5分ごと更新)          │
└──────────────────────────────────────────────┘
                 │ fetch
                 ▼
              iPhone PWA
```

### 3.2 採用理由（GitHub Actions + 静的JSON）

- **既存アーキテクチャと完全整合**: 既存タクシー乗務アプリも GitHub Pages 静的配信
- **無料**: Public repo なので GitHub Actions は時間制限なし、ODPTも無料
- **トークン秘匿**: アクセストークンは GitHub Secrets に格納、リポジトリには含まれない
- **CORS問題なし**: ブラウザは同一オリジンの `data/arrivals.json` を読むだけ
- **遅延の許容性**: 着陸→タクシー乗り場到着は20-30分かかる。5分の更新間隔は実用十分

### 3.3 採用しなかったアーキテクチャ

| 案 | 却下理由 |
|---|---|
| クライアント直叩き | アクセストークン露出、ODPT規約抵触の懸念 |
| Cloudflare Workers プロキシ | 既存アーキテクチャと不整合、運用コスト増 |

## 4. 搭乗者数推定ロジック

### 4.1 計算式

```
推定降客数 = 機材座席数 × 路線別平均搭乗率
```

### 4.2 フォールバック階層

```
1. 機材コード判明 + 路線統計あり → 座席数 × 路線別搭乗率
2. 機材コード判明 + 路線統計なし → 座席数 × 0.70（国内線平均）
3. 機材コード null               → "機材不明" 表示、推定値ダッシュ表示
                                   ヒートマップ集計から除外、不明便数を併記
```

### 4.3 実数ではない明示

UIでは必ず「推定」「平均搭乗率〇〇%（路線実績）」を併記し、実数を装わない。
**実数取得は技術的に不可能**であることを README に明記する。

## 8. GitHub Actions ワークフロー

### 8.1 `update-arrivals.yml`

- **トリガー**: cron `*/5 * * * *`（5分ごと、JST 5:00-24:00 のみ）
- **実行**: `node scripts/fetch-arrivals.mjs`
- **副作用**: `data/arrivals.json` を更新 → commit & push（変更がある場合のみ）
- **Secret**: `ODPT_TOKEN`

### 8.2 commit ログ抑制

- ワークフロー側で「直前と同一内容なら commit しない」
- 別途 `arrivals-data` ブランチへ push する案も検討（main履歴を汚さない）

### 8.3 エラー処理

- ODPT APIタイムアウト: 当該ランをスキップ（前回JSONを保持）
- HTTPエラー: ログ出力のみ、commit せず
- パース失敗: 同上

## 13. 確認事項（リスク・前提）

1. **ODPTのカバー範囲**: SKY / ADO / SNA / JJP / SFJ など LCC が含まれるかは登録後に実データで要確認。含まれない場合は表示が「JAL/ANA便のみ」と注記される
2. **`aircraftModel` の null 率**: 実データ取得後に確認。null率が常時 >50% なら、便番号→機材の補助マッピング（過去データ）を追加検討
3. **路線別搭乗率の精度**: 国交省統計は路線レベル。実際は便ごとに変動するが、本アプリの用途（参考情報）には十分
4. **commit 量**: 5分×288回/日 = 大量commit。`arrivals-data` ブランチ分離 or 同一内容スキップで対応
5. **時刻ゾーン**: ODPT は JST 前提だが、応答にタイムゾーン情報があるかは要確認

## 14. 免責

- 表示する搭乗者数は **推定値** であり、実際の搭乗者数ではない
- 実際の搭乗者数は航空会社の機密情報であり、外部から取得不可能
- 推定根拠（座席数・路線別平均搭乗率）はUIで明示する
- ODPTデータの正確性・遅延については ODPT 規約に従う
